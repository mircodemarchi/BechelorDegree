Hints for Task 1

This document contains hints and guidelines for solving Task 1 of the project task.

This is a warm up task that involves implementing true, false, multiplication, division, AND, OR, not, and negation.

Overview

The files you need to modify to solve task 1 are: Lexer.lex, Parser.grm, Interpreter.sml, TypeChecker.sml, and CodeGen.sml.

Lexer.lex is a Moscow ML Lex program, and Parser.grm is a Moscow ML Yacc program. These programs can contain SML code, but are special languages for lexing and parsing, respectively. The best way to learn the respective languages is to examine the handed out code and study the (perhaps somewhat terse) relevant chapters of the Moscow ML Owner’s Manual.

Boolean Literals: true and false

It might be easiest to start by implementing the boolean literals: true and false. The lexer should be able to recognize these words as tokens. The lexer is not yet able to do so. How do you think that is fixed? Look in Lexer.lex to see how keywords are lexed in general.

In Parser.grm the tokens that are used in the lexer (TRUE and FALSE) must be declared. In the syntax tree, no information other than its type (BoolVal true or BoolVal false) is tied to a boolean literal, unlike e.g. a number token or a string literal token, which both have extra "bagage" tied to them: respectively an integer and a string.

The syntax tree can be found in Fasto.sml. This file should NOT be modified.

But the parser must be able to report its position which is a line number and a column number. So what type should the boolean tokens have?

Further down the file are the production rules for the grammar of Fasto. This production starts with the non-terminal Prog. Just like an expression (Exp) can be a number or an ID (variable name), an expression can also be a boolean expression. The SML expressions within the curly braces are built from constructors found in the syntax tree from Fasto.sml.

Now consider the function evalExp in Interpreter.sml. The first pattern (or case if you will) is Constant (v,_) vtab, ftab) = v. This will match any constant, including true and false. This is because the compiler already supports evaluating (among other things) integer constants, and boolean constants are located in the same part of the syntax tree type, so this automatically works.

But in order to add multiplication, division, AND, etc., Interpreter.sml will need to be changed. You can postpone implementing this until you are done with true and false.

The last thing to change in order to let Fasto handle the boolean literals is the file CodeGen.sml. This is the file that translates an abstract syntax tree into MIPS assembly. Look at the case for handling IntVal for inspiration. The important question is how to model booleans, since MIPS doesn’t have a "boolean type". The simplest way is to express a boolean value as a single byte. There are only two boolean values, and there are 256 different bytes, and you need to find a way to express a boolean as a byte. How would you do that?

Multiplication and Division

Next up is multiplication and division. In the file Lexer.lex, you can look at + and - in the rule Token expression. This might give you an idea of how to represent multiplication and division. The abstract syntax names used to represent multiplication should be reused in Parser.grm.

In Parser.grm the syntax and associativity of multiplication and division must also be defined. In the production rules, these operations must also be added.

In Interpreter.sml, a case of the function evalExp must be added to handle multiplication. Likewise for division.

TypeChecker.sml and CodeGen.sml should also be changed. Again, you can look at Plus and Minus to see how this is done.

AND and OR

Now add AND and OR in the same order as above. Both functions must be short-circuiting. Look at the lecture slides from the first lecture if you are unsure about how to implement short-circuiting or what it means. To implement AND and OR in CodeGen.sml, start by looking at Plus and Minus and understand how they work. Then think about how you would implement the logic of a (short-circuited) AND operation through MIPS instructions. You will probably need to use code such as Mips.BEQ, Mips.J and others.

Not and Negation

not and negate are unary operators working on booleans and integers, respectively. These two instructions should also be implemented. In the interpreter, an error must be raised if the expression following one of these operators does not have the expected type — for example, you cannot negate a string.
